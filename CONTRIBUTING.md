# Contributing

Thanks for improving this playbook.

## Principles

- Be concrete over generic.
- Explain trade-offs, not only happy paths.
- Add operational details (failure handling, observability, scaling limits).
- Keep examples vendor-neutral when possible.

## Pull Request Checklist

- Add or update docs in the right category.
- Include at least one scenario where the pattern should not be used.
- Include one monitoring/alerting recommendation.
- Keep language simple and interview-friendly.
- Link related files in the PR description.

## Writing Format

Use this order for each topic:

1. Problem statement
2. Design options
3. Recommended default
4. Trade-offs
5. Failure modes
6. Metrics and SLO hints
7. Interview answer sketch
